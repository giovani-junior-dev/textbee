Purpose

Transformar o painel (fork self-hosted do TextBee) em produto de marca própria Wablast SMS, com interface em português (PT-BR) por padrão e capacidade de alternar idioma, sem quebrar a base de código nem dificultar futuros git pull do upstream. A i18n precisa ser arquitetura, não gambiarra de strings traduzidas no meio do JSX.

Problem

Hoje os textos estão hardcoded em inglês espalhados por ~105 componentes (79 client). Não há camada de i18n. Pior: o next.config.js usa o bloco i18n legado do Pages Router (locales:['en-US']) — que é incompatível com i18n no App Router e precisa ser removido. Traduzir "na mão" arquivo por arquivo gera dívida: sem fonte única, sem troca de idioma, retrabalho a cada tela nova e conflitos no merge com o upstream.

Solution

Adotar next-intl no modo sem prefixo de locale na URL (locale resolvido por cookie NEXT_LOCALE, default pt-BR). As rotas (/dashboard, /login...) não mudam — zero refactor de roteamento, ideal para um painel interno (não-SEO). Textos viram chaves t('ns.chave') resolvidas de dicionários JSON (messages/pt-BR.json, messages/en.json) com namespaces por feature. Migração incremental tela por tela, com commits por bloco. Junto, o rebranding (cores + nome/logo Wablast SMS) entra como fase isolada de baixo risco.

Questionables

Manter EN além de PT-BR, ou só PT-BR? RESOLVIDO

Decidido (user 2026-06-27): manter EN + PT-BR juntos com toggle. PT-BR default, EN via switcher. Confirma a Fase 5 (language-switcher) como parte do escopo.

Locale por cookie (sem prefixo) vs rota /[locale]/?

Decisão: cookie. O painel é interno (não-SEO), e /[locale]/ exigiria mover todas as rotas app/(app)/ para app/[locale]/(app)/ — refactor grande e mais conflito com upstream. Cookie entrega troca de idioma com risco mínimo.

Tradução das mensagens de erro vindas da API (backend EN)

Assunção: mapear apenas as mensagens mais comuns (login inválido, email não verificado, limites) para chaves errors.*, com fallback exibindo a mensagem original. Traduzir 100% exigiria mexer no backend NestJS (fora do escopo desta fase).

Snippet — i18n/request.ts (referência)

import {getRequestConfig} from 'next-intl/server'
import {cookies} from 'next/headers'
import {locales, defaultLocale, LOCALE_COOKIE} from './locales'

export default getRequestConfig(async () => {
  const c = cookies().get(LOCALE_COOKIE)?.value
  const locale = locales.includes(c as any) ? c! : defaultLocale
  return {
    locale,
    messages: (await import(`../messages/${locale}.json`)).default,
  }
})
Atenção upstream: remover o bloco i18n do next.config.js é obrigatório — no App Router ele é legado e conflita com next-intl. Documentar isso no PR para não voltar num merge futuro.
